Understanding AWS Bedrock’s Data Sharing Architecture

Core Components of Bedrock’s Data Flow

  1. Model Providers – Anthropic, Cohere, and Stability AI expose inference endpoints.
  2. Request/Response Pipelines – API calls pass through Bedrock’s front‑end, which appends metadata tags.
  3. Metadata Tagging – Tags include aws bedrock require flags. These dictate whether the provider may retain the payload for model training.

Bedrock isolates customer data in a dedicated VPC endpoint. Traffic never touches the public internet.

Regulatory Drivers

  • GDPR requires explicit consent before personal data leaves the EU.
  • CCPA mandates opt‑out rights for California residents.
  • AI‑specific statutes (e.g., EU AI Act) impose transparency obligations on training data.

The require sharing data flag aligns Bedrock’s behavior with these mandates. It forces providers to reject any request lacking proper consent.

Interaction With Third‑Party Providers (e.g., Anthropic)

  • Data with Anthropic contracts stipulate that Bedrock only forwards data when the aws bedrock require flag is true.
  • For mythos and future models, Anthropic offers a “no‑learning” option that respects the bedrock require sharing parameter.

Providers publish Service Level Agreements (SLAs) that quantify how require sharing data affects latency and throughput.

Configuring “Require Sharing” Policies in AWS Bedrock

Since the architecture is set, how do you actually implement the controls?

Enabling the RequireSharing Parameter via Console

  1. Open the AWS Management Console $\rightarrow$ Bedrock $\rightarrow$ Settings.
  2. Select Data Sharing from the left‑hand menu.
  3. Toggle RequireSharing to Enabled and select the default consent level.
  4. Click Save.

UI Checklist:

  • Match the Region dropdown to your data‑residency policy.
  • Confirm the IAM Role has bedrock:EnableSharing permissions.
  • Check the Audit Log for recent changes (see [INTERNAL_LINK]).

Programmatic Configuration Using AWS SDKs (Python, Java, Go)

Use this Python snippet to set the aws bedrock require flag when invoking an Anthropic model:

import boto3

client = boto3.client('bedrock-runtime', region_name='us-east-1')
response = client.invoke_model(
    modelId='anthropic.mythos-v1',
    contentType='application/json',
    accept='application/json',
    body='{"prompt":"Explain quantum entanglement"}',
    customAttributes={'RequireSharing': 'true'}  # &lt;-- aws bedrock require flag
)
print(response['body'].read())

Error Handling:

  • Catch ClientError and inspect error_code for AccessDeniedException.
  • Use exponential back-off for ThrottlingException.

Java and Go examples follow this pattern; see the [EXTERNAL_LINK: AWS SDK Documentation].

Validating Policy Enforcement with CloudTrail & Config Rules

  • CloudTrail logs every InvokeModel call, including the RequireSharing attribute.
  • AWS Config rule bedrock-require-sharing-enabled evaluates compliance in near real‑time.

Automated Alert Workflow:

  1. Create a Config rule to check the RequireSharing parameter.
  2. Attach an EventBridge rule to trigger an SNS topic on non‑compliant events.
  3. Deploy a Lambda function to auto-remediate by re‑enabling the flag.

This creates a closed‑loop system that detects and corrects drift automatically.

Frequently Asked Questions

1. How does “aws bedrock require” differ from standard IAM permissions?
IAM controls who can call APIs. The aws bedrock require flag determines what data the provider can use. It is a data-centric governance layer that IAM cannot replace.

2. Can I override the “require sharing” setting for a single request?
No. The flag is enforced at the service level. Bypassing it triggers a 403 Forbidden error, ensuring total compliance.

3. What happens to data processed before enabling “require sharing”?
Historical data follows the original terms. Request a retroactive purge via a support ticket with the provider and document the request for your auditors.

4. Is “data with anthropic” automatically encrypted?
Yes. Bedrock uses TLS 1.2+ for transit and SSE‑KMS for data at rest in S3. This meets most industry standards.

5. How will Mythos‑2 affect existing sharing policies?
Future models will use the same require sharing schema but may add flags for fine-grained consent. Update your Terraform modules as release notes arrive.
